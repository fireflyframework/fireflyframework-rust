# Gap report: pyfly `idp`

Verdict: **partial** · est. 2150 LOC

## What it is
pyfly.idp is the hexagonal identity-provider module: a 19-method async IdpAdapter port (user CRUD, login/logout/refresh/introspect, password change/reset, TOTP MFA challenge/verify, self-registration, role assign/revoke/list) with four adapters and an /idp REST controller. Crucially, all four adapters are REAL implementations, not stubs: internal-db (in-memory, bcrypt + pyotp TOTP MFA + role catalogue), Keycloak (httpx against admin REST + OIDC token/introspect/userinfo endpoints with admin-token TTL caching), Azure AD/Entra (Microsoft Graph v1.0 + ROPC token endpoint, groups-as-roles), and AWS Cognito (boto3 cognito-idp with SECRET_HASH computation, USER_PASSWORD_AUTH/REFRESH_TOKEN_AUTH, global_sign_out, group ops). Only vendor mfa_challenge/mfa_verify deliberately raise NotImplementedError because those providers run MFA server-side. Python-runtime machinery excluded from the gap: @auto_configuration/@bean/@conditional_on_* DI decorators and lazy ImportError-guarded optional imports (httpx/boto3/pyotp/bcrypt).

## pyfly public surface
IdpAdapter protocol (create_user, get_user, find_by_username, update_user, delete_user, list_users, login, logout, refresh, introspect, change_password, reset_password, mfa_challenge, mfa_verify, get_user_info, register_user, get_roles, assign_role, revoke_role, list_roles); DTOs IdpUser, IdpRole, LoginRequest (with mfa_code), PasswordChangeRequest, MfaChallenge, AuthResult (user + tokens + mfa_required + mfa_challenge), SessionIntrospection; adapters InternalDbIdpAdapter (+enable_mfa, create_roles), KeycloakIdpAdapter, AzureAdIdpAdapter, AwsCognitoIdpAdapter; IdpController mounting POST /idp/login|/refresh|/logout|/introspect and /idp/admin/users CRUD + role endpoints; IdpAutoConfiguration selecting the provider from pyfly.idp.provider config.

## Missing in Rust
Rust has the port (crates/idp, 445 LOC), a Full internal-db adapter (752 LOC, bcrypt + HS256 JWT, Go-parity wire shapes), and three vendor crates — but the vendor crates (idp-keycloak, idp-azure-ad, idp-aws-cognito) are explicit contract-only stubs whose every method returns ERR_NOT_IMPLEMENTED, while pyfly's are production implementations. Additionally the Rust Adapter trait covers only 8 of pyfly's 19 operations: missing logout, introspect, find_by_username, list_users, change_password, reset_password, register_user, get_user_info, mfa_challenge/mfa_verify, and all role management (assign_role, revoke_role, list_roles, get_roles); missing models Role, MfaChallenge, SessionIntrospection, and an AuthResult-style mfa_required signal (Rust Token cannot express an MFA-pending login). The internal-db crate has no TOTP MFA, no role catalogue, no token revocation (logout is impossible — JWTs are stateless). No /idp HTTP controller exists anywhere in the workspace (samples and starter crates contain none), and there is no config-driven provider selection equivalent to IdpAutoConfiguration.

## Porting plan
Extend existing crates; no new crate needed. (1) crates/idp: widen the Adapter trait (or add an extension trait ExtendedAdapter to avoid breaking Go-parity consumers) with logout, introspect -> SessionIntrospection, find_by_username, list_users(limit), change_password, reset_password, register_user, mfa_challenge -> MfaChallenge, mfa_verify -> Token, assign_role/revoke_role/list_roles/get_roles; add Role, MfaChallenge, SessionIntrospection serde structs and an Error::MfaRequired(MfaChallenge) variant to model pyfly's mfa_required login outcome (~250 LOC). (2) crates/idp-internal-db: add role catalogue, opaque-token registry for logout/introspect, and TOTP MFA implemented with workspace hmac + sha2 + base64 (RFC 6238 is ~60 LOC; avoids a new totp dep) (~350 LOC). (3) crates/idp-keycloak: real implementation with workspace reqwest — admin client_credentials token cached with expiry margin (Instant-based, mirroring pyfly's monotonic deadline), user CRUD against /admin/realms/{realm}/users, password-grant login, refresh, token introspect, reset-password, realm role-mappings; keep mfa_* returning a sentinel as pyfly does (~450 LOC). (4) crates/idp-azure-ad: reqwest against graph.microsoft.com/v1.0 + login.microsoftonline.com (ROPC login, /me introspect/userinfo, passwordProfile patch, groups-as-roles via /memberOf and /groups/{id}/members/$ref) (~400 LOC). (5) crates/idp-aws-cognito: add aws-sdk-cognitoidentityprovider + aws-config to the workspace catalog (boto3's analog; SECRET_HASH = HMAC-SHA256(secret, username+client_id) base64, already expressible with workspace hmac/sha2/base64) (~450 LOC). (6) /idp router: feature-gated axum Router in crates/idp (or crates/web) taking Arc<dyn Adapter>, mirroring the pyfly route table (~250 LOC). Idiom adaptations: pyfly's DI auto-configuration becomes an explicit from_config(provider: &str, cfg) -> Arc<dyn Adapter> factory in starter-application; lazy optional imports become Cargo features; asyncio.Lock state becomes RwLock as internal-db already does; PermissionError maps to Error::InvalidCredentials. Test fixtures should follow the existing behavior-test pattern (pyfly asserts outbound request shape) using a mock HTTP server (wiremock or hand-rolled hyper).
