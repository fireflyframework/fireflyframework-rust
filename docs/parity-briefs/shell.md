# Gap report: pyfly `shell`

Verdict: **gap** · est. 1500 LOC

## What it is
Spring Shell-style CLI command framework: classes marked @shell_component expose @shell_method commands (with @shell_option/@shell_argument metadata, availability checkers, and type-hint-driven param inference) that the ApplicationContext wires into a ShellRunnerPort; a full Click adapter provides batch dispatch (exit_code+output), command groups, and a line-based interactive REPL. It also ports Spring Boot's CommandLineRunner/ApplicationRunner/ApplicationArguments post-startup hooks, plus conditional auto-configuration (pyfly.shell.enabled). A related but separate `pyfly shell` dev command (pyfly/cli/shell.py) embeds a live Python eval REPL (ptpython/IPython) over the booted context — that part is Python-runtime-specific (runtime code eval) with no meaningful Rust analog.

## pyfly public surface
shell_component (stereotype), shell_method(key, help, group), shell_option(name, type, is_flag, help, default, choices), shell_argument(name, type, help, required, default), shell_method_availability(checker); ShellParam, CommandResult(output, exit_code, is_success), MISSING sentinel; infer_params(func) -> list[ShellParam]; ShellRunnerPort protocol {register_command(key, handler, help_text, group, params), run(args) -> int, run_interactive()}; ClickShellAdapter (full impl: groups, str/int/float/bool coercion, flags, required options, positionals, invoke() -> (exit_code, output), REPL); ApplicationArguments {source_args, option_args, non_option_args, from_args, contains_option, get_option_values}; CommandLineRunner / ApplicationRunner protocols (async run, invoked ordered after context start); ShellAutoConfiguration (conditional on pyfly.shell.enabled + missing ShellRunnerPort bean). Context-side wiring lives in ApplicationContext._wire_shell_commands / _invoke_runners and PyFlyApplication.run (argv dispatch, REPL fallback).

## Missing in Rust
Everything: no crate in fireflyframework-rust provides CLI command registration/dispatch, an argument/option parser with typed coercion and groups, command availability gating, a CommandResult model, an interactive REPL, ApplicationArguments parsing, or ordered CommandLineRunner/ApplicationRunner post-startup runners receiving argv. No clap/rustyline/reedline in the workspace catalog. firefly-lifecycle's ordered on_start hooks cover only the timing aspect of runners, not the args abstraction or shell itself. (The Go port — the Rust port's declared parity baseline — also lacks this module, so this is a pyfly-only addition.) Excluded as Python-specific: the `pyfly shell` Python-eval REPL (pyfly/cli/shell.py) and the reflective type-hint param inference machinery.

## Porting plan
New crate firefly-shell (crates/shell), no new external deps needed (stdlib + tokio from the catalog; skip clap to honor the workspace's minimal-deps policy — pyfly's Click usage is small enough to hand-roll). Key types: (1) ValueType enum {Str, Int, Float, Bool} + ShellParam {name, value_type, is_option, is_flag, required, default: Option<Value>, help, choices: Option<Vec<String>>} and CommandResult {output, exit_code, is_success()}; (2) CommandSpec builder replacing decorators+inference (Command::new("greet").help(..).group(..).arg(..).option(..).flag(..).availability(|| Availability::Unavailable(reason))) — decorator metadata becomes explicit builder calls, availability checker string becomes a closure evaluated at dispatch; (3) handler type Arc<dyn Fn(CommandArgs) -> BoxFuture<'static, Result<String, ShellError>> + Send + Sync> where CommandArgs gives typed getters (get_str/get_i64/get_f64/get_bool); (4) ShellRunner async_trait (register_command/run/run_interactive) with a default StdShell impl: tokenizer + parser supporting positionals, --opt value, --opt=value, flags, choices validation, required-option errors (exit 2 usage / 1 error / 0 ok), grouped help output, and a stdin line-loop REPL; (5) ApplicationArguments struct porting from_args/contains_option/get_option_values exactly per tests/shell/test_runner.py (exact-or-prefix matching, multi-values, values containing '='); (6) CommandLineRunner + ApplicationRunner async traits and an ordered RunnerRegistry — DI scanning becomes explicit registration, invoked from firefly-lifecycle Application::on_start (or a shell::run_app(shell, args) entry mirroring PyFlyApplication.run with REPL fallback). Auto-configuration maps to a feature-gated constructor reading firefly.shell.enabled via firefly-config. Optionally a rustyline feature later; not required for parity (pyfly's REPL is plain input()). Tests ported 1:1 from tests/shell/ (~1,300 py lines): param mapping, parser semantics, groups, availability, runners, ApplicationArguments.
