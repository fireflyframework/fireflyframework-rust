# Gap report: pyfly `session`

Verdict: **gap** · est. 1100 LOC

## What it is
Server-side HTTP session management: an HttpSession wrapper (attribute CRUD, id rotation for anti-session-fixation, invalidation, modified-tracking), a SessionStore port with FULL in-memory (TTL-evicting) and Redis (JSON-serialized, key-prefixed) adapters, and a SessionFilter middleware that loads the session from a cookie (PYFLY_SESSION, HttpOnly, SameSite=Lax, sliding Max-Age, auto-Secure via X-Forwarded-Proto), persists changes post-response, and migrates/deletes entries on rotation/invalidation. It also ports Spring Security's maximumSessions: a SessionRegistry port (FULL memory, Redis sorted-set, and Postgres table adapters) plus a SessionConcurrencyController with evict-oldest/reject-new strategies, all wired via three conditional auto-configurations. Python-specific machinery with no Rust analog: AutoConfiguration.is_available() import-probing, lazy driver imports in adapters, and the importlib-based tagged-dataclass rehydration allowlist (allow_session_type) in the Redis store — in Rust, serde-typed attribute access subsumes this safely.

## pyfly public surface
HttpSession (id, previous_id, is_new, created_at, last_accessed, invalidated, modified, get/set/remove_attribute, get_attribute_names, rotate_id, invalidate, get_data); SessionStore protocol (get/save/delete/exists with TTL); InMemorySessionStore; RedisSessionStore (+ allow_session_type); SessionFilter(store, cookie_name="PYFLY_SESSION", ttl=1800, secure=False) as an ordered OncePerRequestFilter; SessionRegistry protocol (register/deregister/list_sessions oldest-first/count); InMemorySessionRegistry; RedisSessionRegistry; PostgresSessionRegistry; ConcurrencyControlPolicy(max_sessions=-1, strategy="evict-oldest"|"reject-new"); SessionConcurrencyController(on_login -> bool, on_logout); auto-configs keyed on pyfly.session.enabled / pyfly.session.concurrency.enabled with properties store, redis.url, cookie-name, ttl, cookie.secure, concurrency.max-sessions/strategy/registry.

## Missing in Rust
Everything. No session crate exists; "session" appears nowhere in any crate source or docs/DESIGN.md. There is no cookie parsing/issuing anywhere (the only "cookie" hit is a redacted-header name in firefly-web's PII masker), no session object, no session store, no session middleware, and no per-principal concurrency control. firefly-security only does Bearer-token auth + path RBAC; firefly-web has correlation/idempotency/pii/problem layers only. firefly-cache offers a TTL byte-store Adapter (memory/noop/fallback) that could back sessions, but no Redis or Postgres adapter exists in the workspace (no redis/sqlx in the dependency catalog), so the Redis session store and Redis/Postgres registries have no foundation either.

## Porting plan
New crate crates/session -> firefly-session, following the firefly-web tower-Layer idiom (model on crates/web/src/correlation.rs and security/src/filter_chain.rs). Key types: (1) SessionInner { id, previous_id: Option<String>, is_new, invalidated, modified, created_at/last_accessed (chrono), data: HashMap<String, serde_json::Value> } behind a cloneable SessionHandle = Arc<Mutex<SessionInner>> inserted into request extensions so handlers (axum Extension<SessionHandle> or a Session extractor) and the layer share state; methods attribute::<T: DeserializeOwned>, set_attribute(impl Serialize), remove_attribute, attribute_names, rotate_id (uuid v4 simple-hex), invalidate — serde typing replaces the Python type-tag allowlist. (2) #[async_trait] pub trait SessionStore: Send + Sync { get -> Result<Option<HashMap<String,Value>>>, save(id, data, ttl: Duration), delete, exists } + MemorySessionStore (tokio Mutex + Instant expiry) + CacheSessionStore bridging firefly_cache::Adapter (JSON bytes) so any future cache backend works; a direct RedisSessionStore/RedisSessionRegistry requires adding redis = { version = "0.27", features = ["tokio-comp"] } to the workspace catalog behind a `redis` feature (none exists today); PostgresSessionRegistry likewise needs sqlx behind a `postgres` feature — ship memory + cache-bridge first, gate backends. (3) SessionLayer/SessionService<S> (tower Layer + Service): parse Cookie header (hand-rolled ~40 LOC, no cookie crate in catalog), load-or-create, run inner, then persist (delete previous_id on rotation, delete on invalidate, save if modified) and append Set-Cookie with HttpOnly; SameSite=Lax; Max-Age=ttl, Secure auto from x-forwarded-proto or scheme; delete-cookie (Max-Age=0) on invalidation. (4) concurrency module: trait SessionRegistry, MemorySessionRegistry, ConcurrencyPolicy { max_sessions: i32, strategy: Strategy } with enum Strategy { EvictOldest, RejectNew }, SessionConcurrencyController holding Arc<dyn SessionRegistry> + optional deleter closure/Arc<dyn SessionStore>, on_login/on_logout reproducing oldest-first eviction. (5) SessionConfig deserialized via firefly-config (firefly.session.*) with SessionLayer::from_config(cfg, Arc<dyn SessionStore>) replacing the @auto_configuration/@conditional_on_property beans (DI -> explicit builder, the established workspace pattern). Workspace deps suffice except redis/sqlx: tokio, async-trait, axum, tower, http, serde/serde_json, uuid, chrono, thiserror, tracing, firefly-cache, firefly-config. Port the 29-test contract (rotation migrates store+cookie, secure-over-forwarded-proto, expired-entry eviction, evict-oldest deletes store entry, reject-new, logout deregisters).
