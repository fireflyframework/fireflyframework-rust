# Gap report: pyfly `ecm`

Verdict: **partial** · est. 1600 LOC

## What it is
pyfly.ecm is the Enterprise Content Management module: hexagonal ports for document blob storage, document metadata, folders, and e-signature, with DocumentService/ESignatureService orchestrators, multi-version document blobs (DocumentVersion list, per-version download/delete), in-memory metadata/folder repos, and seven adapters that are ALL full implementations — local filesystem, AWS S3 (boto3), Azure Blob (azure-storage-blob), DocuSign (REST v2.1), Adobe Sign (REST v6), Logalty (REST), and a NoOp e-signature for tests. An EcmAutoConfiguration selects storage/e-signature providers from YAML (pyfly.ecm.storage.provider / pyfly.ecm.esignature.provider). Python-runtime machinery with no Rust analog: lazy `import boto3/httpx` ImportError hints, run_in_executor offloading of sync SDKs, asyncio.Lock, and the @auto_configuration/@bean DI decorators themselves (the config-driven selection semantics do have a Rust analog as a factory).

## pyfly public surface
Models: Document, DocumentVersion, Folder, Recipient, SignatureRequest, ESignatureEnvelope, ESignatureStatus (DRAFT/SENT/SIGNED/DECLINED/EXPIRED). Ports: DocumentStoragePort (upload/download/delete, version-aware), MetadataStoragePort (save/get/list(folder_id, limit)/delete), FolderRepositoryPort (save/get/list/delete), ESignatureAdapter (send/get/cancel). Services: DocumentService (upload/download/list/delete with partial-delete result semantics/create_folder), ESignatureService (request/get/cancel). Adapters: LocalFilesystemStorageAdapter, AwsS3StorageAdapter, AzureBlobStorageAdapter, DocuSignESignatureAdapter, AdobeSignESignatureAdapter, LogaltyESignatureAdapter, NoOpESignatureAdapter; InMemoryMetadataStorage, InMemoryFolderRepository; EcmAutoConfiguration.

## Missing in Rust
Rust crates/ecm covers the core (ContentStore, DocumentService, ESignatureProvider ports; Document/Folder/SignatureRequest/SignatureStatus models with wire-parity serde; LocalStore full; in-memory Service with checksum extras). Missing vs pyfly: (1) All 5 vendor adapters are sentinel stubs ("firefly/<module>: not yet implemented") while pyfly ships full, behavior-tested implementations: AWS S3 put/get/delete with <prefix>/<docId>/v<n> keys, Azure Blob upload/download/delete, DocuSign envelopes create/get/void with status mapping, Adobe Sign agreements create/get/state-cancel, Logalty envelopes POST/GET/DELETE — this is the headline gap (~1,150 LOC of real integration behavior in pyfly). (2) Folder operations: Rust has the Folder model but no FolderRepository port, no in-memory impl, no create_folder. (3) Document listing by folder (MetadataStoragePort.list(folder_id, limit)) — Rust Service has no list at all. (4) Multi-version blobs: pyfly DocumentVersion list with per-version download/delete; Rust Document.version is pinned to 1 and ContentStore keys carry no version. (5) No shipped NoOp e-signature provider (test-internal StaticSigner only). (6) No config-driven provider selection factory (pyfly EcmAutoConfiguration semantics). Note: the Rust stubs deliberately mirror the Go port (also stubbed), but a consumer migrating pyfly code would find the vendor functionality absent.

## Porting plan
Extend existing crates; no new crate needed. (a) crates/ecm: add FolderRepository trait + InMemoryFolderRepository; add list(folder_id, limit) to DocumentService (or a separate MetadataStore trait) and implement in Service; optionally add DocumentVersion + version-aware keys (<id>/v<n>) in Service/LocalStore; add a public NoOpESignature provider (signs immediately, in-memory envelope map behind tokio RwLock); add a from_config factory mapping firefly.ecm.storage.provider / esignature.provider strings to boxed trait objects (the DI-free analog of EcmAutoConfiguration), using firefly-config. ~350 LOC. (b) crates/ecm-esignature-docusign / -adobe-sign / -logalty: replace sentinel bodies with real REST calls using workspace reqwest (already in the dep catalog: reqwest 0.12 with json); port pyfly's request payloads, status-mapping tables (DocuSign created/sent/delivered/completed/declined/voided/expired; Adobe OUT_FOR_SIGNATURE/SIGNED/CANCELLED/...; Logalty DRAFT/SENT/PENDING/...) onto SignatureStatus; keep Config structs and ERR_NOT_IMPLEMENTED constants for back-compat; test against an in-process axum mock server (workspace pattern). ~220 LOC each, ~650 total. (c) crates/ecm-storage-aws: add aws-sdk-s3 + aws-config to the workspace catalog (not present today) and implement put/get/delete with ByteStream <-> ContentReader bridging; honor Config.endpoint for LocalStack/MinIO tests. ~280 LOC. (d) crates/ecm-storage-azure: add azure_storage_blobs (or implement Blob REST + SharedKey signing over reqwest using existing hmac/sha2/base64 deps if avoiding the unofficial SDK); same key scheme. ~320 LOC. Idiom adaptations: pyfly's lazy SDK imports become Cargo features/optional deps; run_in_executor disappears (native async SDKs); decorator-based auto-config becomes the explicit from_config factory; bool-returning delete maps to Result with the existing EcmError::NotFound/Provider sentinels.
