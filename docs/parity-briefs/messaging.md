# Gap report: pyfly `messaging`

Verdict: **partial** · est. 1350 LOC

## What it is
pyfly.messaging is the framework's low-level message-broker abstraction (distinct from pyfly.eda's event-envelope layer): a hexagonal MessageBrokerPort (publish topic+bytes+key+headers; subscribe with optional consumer group; start/stop lifecycle) with three fully working adapters — InMemoryMessageBroker (broadcast to ungrouped subscribers, one-delivery-per-group with round-robin within a group), KafkaAdapter (aiokafka producer + per-(topic,group) consumers with per-message error isolation and late-subscription support), and RabbitMQAdapter (aio-pika robust connection, durable direct exchange, topic→routing-key, group→queue-name). On top it provides @message_listener(topic, group, retries, retry_delay, dead_letter_topic) and wrap_listener(), an adapter-agnostic retry (linear backoff) + dead-letter wrapper that republishes exhausted messages with x-original-topic/x-exception headers (mirrors Spring Kafka @RetryableTopic). Python-runtime-specific and excluded from the gap: the decorator-metadata + ApplicationContext bean-scan wiring (Rust analog is explicit subscribe at startup) and MessagingAutoConfiguration's provider auto-detection by probing installed libraries (aiokafka/aio_pika) — the Rust analog is cargo features + explicit config.

## pyfly public surface
Message (topic, value: bytes, key: bytes|None, headers: dict[str,str]); MessageHandler = async Callable[[Message], None]; MessageBrokerPort protocol — publish(topic, value, *, key, headers), subscribe(topic, handler, group=None), start(), stop(); @message_listener(topic, group=None, *, retries=0, retry_delay=0.0, dead_letter_topic=None); wrap_listener(handler, broker, *, retries, retry_delay, dead_letter_topic) -> MessageHandler; adapters: InMemoryMessageBroker, KafkaAdapter(bootstrap_servers), RabbitMQAdapter(url, exchange_name="pyfly"); MessagingAutoConfiguration (provider detection + bean wiring, config keys pyfly.messaging.provider / pyfly.messaging.kafka.bootstrap-servers / pyfly.messaging.rabbitmq.url)

## Missing in Rust
crates/eda covers the abstraction (Publisher/Subscriber/Broker ports, Event envelope, full InMemoryBroker, config-driven factory entry points) but: (1) the Kafka and RabbitMQ transports are deliberate scaffolds — new_kafka_broker/new_rabbitmq_broker return EdaError::KafkaUnavailable/RabbitMqUnavailable sentinels, and no Kafka (rdkafka) or AMQP (lapin) client exists anywhere in the workspace catalog, whereas pyfly's adapters are fully functional; (2) no consumer-group semantics — Subscriber::subscribe(topic, handler) has no group parameter and InMemoryBroker only broadcasts, while pyfly delivers once per group with round-robin among group members; (3) no partition/message key — Event has no key field (pyfly Message.key drives Kafka partitioning); (4) no retry + dead-letter listener wrapper — nothing equivalent to wrap_listener's bounded retries with linear backoff and DLQ republish with x-original-topic/x-exception diagnostic headers (crates/resilience has generic retry but no message-DLQ routing; crates/webhooks' DLQ is webhook-domain-specific).

## Porting plan
Extend crates/eda for the abstraction gaps (~300 LOC incl. tests): add `key: Option<Vec<u8>>` to Event (serialize as base64, omit when None, keeping cross-port wire compat); add group-aware subscription — either `async fn subscribe_group(&self, topic: &str, group: &str, h: Handler)` with a default impl delegating to subscribe, or a SubscribeOptions struct — and implement per-(topic,group) round-robin in InMemoryBroker via AtomicUsize counters; add listener.rs with `pub fn wrap_listener(h: Handler, publisher: Arc<dyn Publisher>, policy: ListenerPolicy) -> Handler` where ListenerPolicy { retries: u32, retry_delay: Duration, dead_letter_topic: Option<String> } — explicit wrapper replacing the Python decorator, tokio::time::sleep(delay * attempt) linear backoff, republish with x-original-topic/x-exception headers, zero-cost pass-through when unconfigured. Then ship the two transport crates the eda README already plans: firefly-eda-kafka (~550 LOC w/ tests; add rdkafka to the workspace catalog; FutureProducer + one StreamConsumer task per (topic,group), per-message error isolation logging and continuing — mirroring pyfly's _consume_loop — late subscribe-after-start spawning its own consumer, graceful close()) and firefly-eda-rabbitmq (~500 LOC w/ tests; add lapin to the catalog; robust reconnecting connection, single durable direct exchange, topic→routing key, group→queue name defaulting to firefly.{topic}, ack-on-success). Replace the sentinel factories with feature-gated real wiring and let starter-core select the broker from a config key (e.g. firefly.messaging.provider with explicit kafka/rabbitmq/memory values — no import auto-detection, which has no Rust analog). Integration tests via testcontainers behind an ignored/feature flag.
