# Gap report: pyfly `context`

Verdict: **partial** · est. 450 LOC

## What it is
pyfly's Spring ApplicationContext equivalent (~2,000 LOC, 10 files): a bean-registry lifecycle manager layered on the DI container that runs conditional auto-configuration (@conditional_on_property/class/bean/missing_bean/single_candidate/expression/resource/web_application, @auto_configuration, two-pass ConditionEvaluator), @bean factory-method processing with topological sort, @post_construct/@pre_destroy with per-bean shutdown timeouts, BeanPostProcessor (AOP weaving) hooks, and decorator-wiring passes for event/message listeners, CQRS handlers, @scheduled, @async_method, shell commands and runners. It also provides a profile-aware Environment (Spring 2.4 boolean profile-expression grammar), a typed in-process ApplicationEventBus/ApplicationEventPublisher with lifecycle events (ContextRefreshed/ApplicationReady/ContextClosed/RefreshScopeRefreshed), Spring Cloud-style ContextRefresher + refresh scope (runtime config reload + bean eviction, exposed via actuator POST /refresh), and a contextvars-backed RequestContext (request_id, security context, attributes). Much of this is Python-runtime machinery (decorator scanning, module-importability conditions, runtime DI, dynamic bean replacement) that the Rust port — following its Go-port precedent of explicit composition — has no meaningful analog for.

## pyfly public surface
ApplicationContext (register_bean, register_post_processor, get_bean, get_bean_by_name, get_beans_of_type, contains_bean, start, stop, container/config/environment/event_bus properties, wiring_counts, get_bean_counts_by_stereotype); Environment (active_profiles, accepts_profiles with &/|/!/() grammar, get_property); ApplicationEvent + ContextRefreshedEvent/ApplicationReadyEvent/ContextClosedEvent/RefreshScopeRefreshedEvent; ApplicationEventBus (subscribe/publish), ApplicationEventPublisher, @app_event_listener; @post_construct, @pre_destroy; BeanPostProcessor protocol (before_init/after_init); ContextRefresher.refresh(); RequestContext (init/current/clear, request_id, security_context, get/set attributes, HTTP_SESSION_KEY); conditional decorators: conditional_on_property, conditional_on_expression, conditional_on_class, conditional_on_bean, conditional_on_missing_bean, conditional_on_single_candidate, conditional_on_web_application, conditional_on_resource, auto_configuration; ConditionEvaluator (should_include, should_include_method).

## Missing in Rust
Most semantics exist in Rust idiom: start/stop lifecycle with ordered start hooks, reverse-order stop hooks, drain budget (= pyfly shutdown-timeout) and failure rollback are in crates/lifecycle (Application); the composition-root/auto-configuration role is crates/starter-core (Core); profile-specific config is crates/config (active_profile + profile_sources); RequestContext is covered by crates/kernel correlation task-locals + crates/web CorrelationLayer/request extensions + crates/security Authentication-in-extensions; in-process pub/sub is crates/eda InMemoryBroker. Genuinely absent for a migrating consumer: (1) runtime configuration refresh — no analog of ContextRefresher/RefreshScope/RefreshScopeRefreshedEvent or the actuator POST /refresh endpoint anywhere (grep confirms zero 'refresh' hits in crates/config, crates/config-server, crates/actuator; Rust config is load-once into a plain struct); (2) a typed in-process application event bus — pyfly publishes arbitrary objects to listeners matched by type (ApplicationEventPublisher/@app_event_listener); Rust offers only topic-keyed eda events and fixed lifecycle hooks, no type-dispatched broadcast; (3) minor: multiple active profiles and the boolean profile-expression grammar ('prod & cloud | qa', '!dev') — Rust supports exactly one FIREFLY_PROFILE string. Excluded as Python-only plumbing: DI container access, decorator scanning/wiring passes, BeanPostProcessor/AOP weaving, @conditional_on_class (Cargo features fill this role), @bean topological sorting, lazy bean init.

## Porting plan
No new crate; extend two existing crates. (A) Runtime refresh — extend crates/config with a refresh module: pub struct ConfigHandle<T>(tokio::sync::watch::Sender/Receiver pair or std::sync::RwLock<std::sync::Arc<T>>) exposing get() -> Arc<T> and subscribe(); pub struct ContextRefresher holding a boxed reload closure (re-runs load_from_profile/source chain) with async fn refresh(&self) -> Result<Vec<String>, ConfigError> returning changed top-level keys (diff via serde_json::Value comparison — serde_json is already in the workspace catalog; no new deps). Then extend crates/actuator with an optional POST /actuator/refresh route (ActuatorConfig gains a refresher: Option<Arc<dyn Refresher>>) returning the changed-keys JSON array, matching the pyfly/Spring Cloud wire shape; wire it through starter-core Core. ~300 LOC incl. tests. (B) Typed app-event bus — extend crates/eda (or starter-core) with AppEventBus: HashMap<TypeId, Vec<Box<dyn Fn(&dyn Any) + Send + Sync>>> guarded by RwLock, publish<E: Any + Send + Sync>(&self, ev: E) and subscribe<E>(handler), mirroring the TypeId-keyed registry idiom already used by crates/cqrs Bus; lifecycle events become plain structs (ApplicationReady, ContextClosed) published by starter-core around Application::run. ~150 LOC incl. tests. (C) Optional: accept comma-separated FIREFLY_PROFILE and an accepts_profiles(expr) evaluator in crates/config (~80 LOC) — recommend deferring unless consumers ask. Idiom adaptations: @refresh_scope beans -> re-read ConfigHandle::get() per call instead of bean eviction; @app_event_listener decorator -> explicit bus.subscribe::<E>(handler) at composition time in main().
