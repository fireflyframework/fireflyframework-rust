# Gap report: pyfly `admin`

Verdict: **gap** · est. 2500 LOC

## What it is
pyfly.admin is an embedded, Spring-Boot-Admin-style management dashboard: a self-contained SPA (vanilla-JS ES modules, ~948KB of static assets, 18 views incl. overview/health/metrics/environment/config/loggers/traces/logfile/caches/cqrs/transactions/scheduled/mappings/runtime/wallboard) mounted at a configurable path, backed by ~25 JSON API endpoints and 7 SSE live streams, with role-guarded access, an HTTP-trace ring-buffer filter, an in-memory log ring buffer, mutation endpoints (set logger level, cache evict/clear, logfile clear), and an AdminViewExtension plugin point. It also ships a multi-instance "server mode" (InstanceRegistry + static discovery + register/deregister REST API) and "client mode" (AdminClientRegistration that self-registers/deregisters with a remote admin server on lifecycle start/stop). Note: several views are Python-runtime introspection with no Rust analog — beans/bean-graph/conditions/autowired (DI container reflection; the Rust workspace has no DI container), Python GC/thread stats in RuntimeProvider, and ASGI server info in ServerProvider — these are excluded from the gap.

## pyfly public surface
Exported: AdminProperties (pyfly.admin.*: enabled, path, title, theme, require_auth, allowed_roles, refresh_interval), AdminServerProperties (pyfly.admin.server.*: enabled, poll_interval, timeouts, instances[]), AdminClientProperties (pyfly.admin.client.*: url, auto_register), AdminViewExtension protocol (view_id/display_name/icon/get_data), AdminViewRegistry (register/get_extensions/discover_from_context). Operational surface consumed by apps: AdminRouteBuilder.build_routes() mounting GET/POST /admin/api/{overview,beans,beans/graph,beans/{name},health,env,config,loggers,loggers/{name}(POST),metrics,metrics/{name},scheduled,mappings,caches,caches/keys,caches/{name}/evict(POST),cqrs,transactions,traces,logfile,logfile/clear(POST),runtime,server,views,settings,instances(GET/POST/DELETE)} + /admin/api/sse/{health,metrics,traces,logfile,runtime,server,beans} + static SPA with version-stamped no-cache serving; TraceCollectorFilter (500-entry HTTP trace ring buffer, excludes /admin/* and /actuator/*); AdminLogHandler (2000-record log ring buffer with incremental get_records(after=id)); InstanceRegistry/InstanceInfo/StaticDiscovery; AdminClientRegistration (register/deregister with start/stop lifecycle hooks); AdminAutoConfiguration wiring it all conditionally.

## Missing in Rust
No admin dashboard exists anywhere in the 52-crate workspace. crates/actuator only serves raw /actuator data (health, Prometheus metrics, redacted OS env, info, tokio task count, version); crates/backoffice is unrelated (X-BackOffice-* header context middleware). Missing for a migrating consumer: (1) the embedded SPA UI and its /admin/api JSON contract; (2) SSE live streams for health/metrics/traces/logs/runtime; (3) HTTP request trace ring buffer (tower-layer equivalent of TraceCollectorFilter) and its /traces endpoint; (4) in-memory log capture ring buffer + /logfile + clear; (5) dynamic logger-level listing/mutation (tracing-subscriber reload — meaningful in Rust, absent); (6) environment/config views with property-source origin attribution and secret masking (actuator /env shows OS env vars only; firefly-config has no property_sources/mask_value surface); (7) cache stats/keys/evict management endpoints (cache::Adapter has clear() but no stats/keys, and no HTTP surface); (8) scheduled-task listing (Scheduler stores tasks but exposes no accessor); (9) CQRS handler listing (cqrs bus registry is private); (10) saga/TCC definition listing from orchestration; (11) route/mappings listing; (12) admin auth guard (require_auth + allowed_roles); (13) multi-instance server mode (InstanceRegistry + register/deregister API + static discovery) and client self-registration on lifecycle start/stop; (14) AdminViewExtension plugin point. Excluded as Python-specific: beans/bean-graph/bean-detail/conditions/autowired introspection (no DI container in Rust), Python GC/thread/interpreter stats, ASGI server info.

## Porting plan
New crate crates/admin (firefly-admin), modeled on actuator's mount() idiom. Key pieces: (a) AdminConfig/AdminServerConfig/AdminClientConfig deserialized via firefly-config; (b) pub fn mount(cfg: AdminConfig, deps: AdminDeps) -> axum::Router — AdminDeps is an explicit wiring struct (DI discovery -> constructor injection): Arc<HealthComposite>, Arc<MetricRegistry> (from firefly-actuator), Arc<Scheduler>, Option<Arc<dyn firefly_cache::Adapter>>, Arc<TraceBuffer>, Arc<LogBuffer>, Option<Arc<InstanceRegistry>>, Vec<Arc<dyn AdminView>>; (c) #[async_trait] trait AdminView { view_id/display_name/icon/data() -> serde_json::Value } + AdminViewRegistry (Protocol -> trait object); (d) TraceLayer: tower Layer with Mutex<VecDeque<TraceEntry>> (cap 500), skip /admin|/actuator paths; (e) LogBuffer: a tracing_subscriber::Layer pushing formatted events into a Mutex<VecDeque> (cap 2000) with monotonic ids for incremental SSE; (f) loggers endpoints backed by tracing_subscriber::reload::Handle<EnvFilter> — set_level becomes an EnvFilter directive rewrite (TRACE..OFF map naturally to tracing levels); (g) SSE handlers reuse firefly-sse Event/SseWriter (or axum Sse) driven by tokio::time::interval streams, one per pyfly stream; (h) server mode: InstanceRegistry (RwLock<HashMap<String, InstanceInfo>>), StaticDiscovery from config, POST/DELETE /admin/api/instances; client mode: AdminClient using reqwest with register()/deregister() hooked into firefly-lifecycle start/stop; (i) embed the existing SPA assets via rust-embed/include_dir and vendor pyfly's JS/CSS nearly verbatim (the API contract is identical), serving index.html with <base href> injection + ?v= version stamping + Cache-Control: no-cache; (j) auth guard as an axum middleware checking firefly-security claims/roles when require_auth is set. Small upstream extensions needed (~150 LOC total): Scheduler::tasks() snapshot accessor in crates/scheduling; a registered-type-names listing on the cqrs bus registry; saga/TCC definition listing in crates/orchestration; optional stats()/keys() (default-empty) on cache::Adapter. Omit beans/bean-graph endpoints (no DI container) — return the AdminView-driven "views" list instead; adapt runtime endpoint to tokio worker/task counts + process RSS (sysinfo) instead of Python GC; drop ServerProvider's ASGI specifics in favor of bind-addr/worker-thread info.
