# Gap report: pyfly `aop`

Verdict: **gap** · est. 550 LOC

## What it is
Spring-style aspect-oriented programming for pyfly: classes marked @aspect declare advice methods (@before, @after_returning, @after_throwing, @after, @around) bound to a dot-segmented pointcut glob language (* = one segment, ** = any depth, partial globs like get_*). An AspectBeanPostProcessor hooks the DI container lifecycle, collects aspects into an AspectRegistry (ordered by @order), and weave_bean monkey-patches matching public methods of every non-aspect bean with sync/async wrappers that run the advice chain around the original call via a JoinPoint (target, args, return_value, exception, proceed). Fully implemented and tested (7 test files incl. end-to-end ApplicationContext integration); the runtime weaving itself (setattr monkey-patching, getattr_static property-skipping, BeanPostProcessor) is Python-runtime machinery with no Rust analog.

## pyfly public surface
@aspect class decorator; advice decorator factories before(pointcut), after_returning(pointcut), after_throwing(pointcut), after(pointcut), around(pointcut); JoinPoint dataclass (target, method_name, args, kwargs, return_value, exception, proceed); matches_pointcut(pattern, qualified_name); AspectRegistry (register / get_all_bindings / get_matching) and AdviceBinding (advice_type, pointcut, handler, aspect_order); weave_bean(bean, qualified_prefix, registry); AspectBeanPostProcessor (before_init / after_init); AopAutoConfiguration providing the post-processor bean, gated by config key pyfly.aop.enabled. Behavior contract: before -> around chain (first-registered outermost, proceed() invokes next) -> after_returning on success / after_throwing on exception (re-raised) -> after always; aspects ordered globally by @order; aspects never woven into themselves; qualified names are stereotype.ClassName.method (e.g. service.OrderService.create).

## Missing in Rust
No AOP/aspect/pointcut/joinpoint code exists anywhere in the Rust workspace (grep over all 52 crates: zero hits). The Rust ports of cross-cutting concerns are boundary-specific, not generic: firefly-web has tower Layers at the HTTP edge (problem/correlation/idempotency/PII), firefly-cqrs has a Middleware trait wrapping DynHandler at the bus dispatch boundary, and firefly-resilience has Decorator::call + Chain for guarded operations. Genuinely absent: (1) the pointcut pattern language and matcher (* / ** / partial-glob segments against qualified names); (2) an ordered advice registry binding the five advice kinds to pointcuts; (3) a generic advice-chain executor (before / around-with-proceed / after_returning / after_throwing / after with the exact ordering and re-raise semantics) applicable to arbitrary service methods rather than one fixed boundary. A pyfly user with @aspect logging/auditing/caching aspects matched by pattern across services has nothing to migrate to. Excluded as Python-specific (not counted as missing): runtime monkey-patch weaving of live instances, property-descriptor skipping, and the BeanPostProcessor/DI-container integration (the Rust workspace has no bean container).

## Porting plan
New crate crates/aop -> firefly-aop (matches the workspace's crate-per-module layout; depends only on workspace-catalog deps: regex, async-trait, futures, thiserror, tokio for tests). Key types: (1) Pointcut — compiled-regex matcher porting _segment_to_regex/_pattern_to_regex 1:1 (pure logic, trivially portable, ~80 LOC); (2) JoinPoint/Invocation — { type_name: &'static str, method_name: &'static str, args: Arc<dyn Any + Send + Sync>, result: Option<Arc<dyn Any + Send + Sync>>, error: Option<&dyn Error> } since Rust has no reflective arg capture; (3) AdviceKind enum + an Aspect trait with default no-op async hooks (before/after_returning/after_throwing/after) plus around(&self, jp, Proceed) -> BoxFuture, where Proceed is a boxed async fn chaining to the next around or the original call (mirror firefly-resilience's Operation/OpFuture types); (4) AspectRegistry { register(Arc<dyn Aspect>, pointcut: &str, order: i32), matching(&str) } with bindings kept order-sorted — explicit order param replaces @order discovery; (5) the chain executor intercept(registry, qualified_name, args, op) enforcing the exact before/around/after_returning/after_throwing/after ordering with error re-propagation. Idiom adaptation for weaving: no runtime reflection, so replace BeanPostProcessor auto-weaving with explicit interception sites — a declarative macro advised!(registry, "service.OrderService.create", (args) => self.create(a, b).await) for call-site wrapping, and optionally a proc-macro #[advised(prefix = "service.OrderService")] on impl blocks that rewrites each pub async fn to route through the registry (registry handle stored in the struct or passed in) — same pattern the workspace already uses for explicit composition (tower layers, Bus::use_middleware). Document in the crate README that HTTP-edge and bus-dispatch cross-cutting should keep using firefly-web layers and firefly-cqrs Middleware; firefly-aop is for pattern-matched service-method advice. Tests port the pyfly contract: pointcut table tests, advice ordering, around result modification, multi-around chaining (first-registered outermost), after_throwing + after on error, multi-aspect ordering by order value, non-matching methods untouched.
