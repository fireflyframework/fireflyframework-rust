# Gap report: pyfly `websocket`

Verdict: **gap** · est. 550 LOC

## What it is
pyfly.websocket provides decorator-driven WebSocket server support for class-based controllers: a framework-agnostic WebSocketSession wrapper over a Starlette WebSocket (accept with subprotocol, send/receive text|json|bytes, close with code/reason, path/query/header access, an `accepted` flag), a WebSocketHandler protocol with lifecycle hooks, and a @websocket_mapping decorator. A Starlette WebSocketRegistrar scans @rest_controller/@controller beans in the DI container, composes @request_mapping base paths, builds WebSocketRoutes with lazy bean resolution, and enforces the lifecycle contract: handler owns the full accept+receive loop; on_disconnect is auto-invoked only if the connection was accepted, with cleanup failures logged not swallowed; on_connect/on_message are explicitly NOT auto-dispatched (locked in by tests/websocket/test_ws_lifecycle.py and test_ws_e2e.py). The decorator metadata (__pyfly_ws_mapping__) and DI bean-scanning registrar are Python-idiom plumbing with no Rust analog — in Rust routes are registered explicitly on axum::Router (as starter-core does).

## pyfly public surface
WebSocketSession (accepted, path_params, query_params, headers; accept(subprotocol), send_text/send_json/send_bytes, receive_text/receive_json/receive_bytes, close(code, reason)); WebSocketHandler protocol (on_connect, on_message — convenience only; on_disconnect — auto-called post-handler when accepted); websocket_mapping(path) decorator; WebSocketRegistrar.collect_routes(ctx) (adapter-internal).

## Missing in Rust
No WebSocket server support exists anywhere in the 52-crate workspace. crates/web is HTTP-middleware-only (problem/correlation/idempotency/pii), crates/sse covers only the half-duplex SSE case, and the WS client in crates/client is an acknowledged placeholder (new_websocket -> ClientError::TransportNotRegistered). The workspace axum dep does not even enable the "ws" feature. A migrating consumer would find absent: (1) the WsSession-style typed send/receive API including JSON convenience methods, (2) the lifecycle guarantee that a disconnect/cleanup hook runs after the handler returns or the socket closes — only for accepted connections — with hook failures logged rather than swallowed or propagated, and (3) graceful handling of client disconnects (disconnect exception swallowed, unexpected handler errors logged at warn).

## Porting plan
New crate firefly-websocket (matches one-module-per-crate layout; sibling of firefly-sse, which already pitches itself as "the half-duplex equivalent of WebSockets"). Build on axum's native WS: add `axum = { workspace = true, features = ["ws"] }`. Key items: (1) `WsSession` wrapping axum::extract::ws::WebSocket with send_text/send_json/send_bytes, recv_text/recv_json/recv_bytes (recv_* skip Ping/Pong, return WsError::Disconnected on Close/None), close(code, reason); (2) `WsError` enum (Disconnected, Protocol, Serde) via thiserror; (3) `WebSocketHandler` async-trait with default no-op on_connect/on_message (documented as convenience, never auto-dispatched — mirror the pyfly contract) and on_disconnect; (4) a route helper `ws_route<H: WebSocketHandler>(handler: Arc<H>) -> MethodRouter` (or `serve_ws(upgrade: WebSocketUpgrade, handler) -> Response`) that performs `ws.on_upgrade`, runs `handler.handle(session)`, logs unexpected errors via tracing::warn, and always invokes on_disconnect afterward, logging its failures — note axum only runs the closure post-handshake, so the "only if accepted" gate is satisfied structurally and the `accepted` flag becomes unnecessary (document this adaptation, like sse's "Adaptation from Go" section). Subprotocol selection maps to WebSocketUpgrade::protocols(); path/query/headers map to axum extractors (Path/Query/HeaderMap) rather than session properties — idiomatic, document the mapping. Decorator + DI registrar are replaced by explicit `Router::new().route("/ws/echo", ws_route(...))` registration, consistent with starter-core's apply_middleware pattern. Workspace deps suffice (axum+ws, tokio, futures, serde/serde_json, tracing, async-trait, thiserror); add tokio-tungstenite as a dev-dependency for e2e echo/lifecycle tests mirroring tests/websocket/test_ws_e2e.py. Optionally, a follow-up could back crates/client's new_websocket placeholder with a tokio-tungstenite transport, but that is the client package's scope, not this one's.
