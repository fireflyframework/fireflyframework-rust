# Pyfly-parity audit gaps (post-P2)

## HIGH

- **[data, cqrs, eventsourcing, transactional]** Workflow step compensation. pyfly's WorkflowExecutor rolls back already-completed compensatable steps in reverse order on any step failure (workflow/executor.py _compensate, @compensation_step). The Rust Workflow is explicitly fail-fast with NO compensation (README: 'Workflow | DAG with parallel branches | None — fail-fast'; workflow.rs short-circuits on first node error and returns). A migrating consumer relying on workflow-level rollback loses it (only Saga compensates in Rust).
  - Where: pyfly src/pyfly/transactional/workflow/executor.py (_compensate) vs rust crates/orchestration/src/workflow.rs

- **[data, cqrs, eventsourcing, transactional]** Advanced workflow wait/compose primitives. pyfly workflow steps support WaitForAll / WaitForAny (gather/race over multiple signals+timers with timeout), ChildWorkflow (start a sub-workflow as a step), ContinueAsNew, conditional steps (condition.py SpEL-substitute skip), and async fire-and-forget steps. The Rust workflow has only single-signal Node::wait_for_signal and single Node::timer; there are no wait-all/wait-any, child-workflow, continue-as-new, conditional, or async-step constructs anywhere in the workspace.
  - Where: pyfly src/pyfly/transactional/workflow/{annotations.py,executor.py,child_workflow_service.py,continue_as_new_service.py,condition.py} vs rust crates/orchestration/src/{workflow.rs,signal.rs,timer.rs}

- **[data, cqrs, eventsourcing, transactional]** Per-step retry/backoff/timeout enforcement. pyfly's StepInvoker applies RetryPolicy (max_attempts, exponential backoff, jitter, per-attempt timeout) to every saga/workflow/TCC step. In Rust the RetryPolicy struct exists in model.rs but is never enforced by any engine — saga.rs/workflow.rs/tcc.rs run each step exactly once (no with_retry, no retry loop, no timeout wrapping). The saga.rs doc comment even claims 'the engine may retry' but no retry code exists.
  - Where: pyfly src/pyfly/transactional/core/step_invoker.py vs rust crates/orchestration/src/{saga.rs,workflow.rs,tcc.rs} (RetryPolicy in model.rs is unused by engines)

- **[data, cqrs, eventsourcing, transactional]** Inter-step data passing / argument injection. pyfly steps declare typed inputs via Annotated markers (Input, FromStep, Variable, SetVariable, Header(s), CorrelationId, FromCompensationResult) resolved from an ExecutionContext (core/argument.py + ArgumentResolver). Rust saga/workflow/TCC steps are zero-arg closures (Fn() -> Fut) with no input, no access to prior step results, no shared variables/headers at runtime. report.rs has variables/StepReport for reporting only, not runtime injection. Migrating a multi-step saga that threads data between steps has no equivalent.
  - Where: pyfly src/pyfly/transactional/core/argument.py vs rust crates/orchestration/src/{saga.rs,workflow.rs,tcc.rs}

- **[data, cqrs, eventsourcing, transactional]** Global cross-aggregate event stream. pyfly EventStore exposes stream_all(after_event_id, limit) and ProjectionRunner polls it cursor-style (at-least-once, in-order) to build read models from the entire event log. Rust EventStore trait has only per-aggregate load/load_after; ProjectionRunner::replay re-applies a single aggregate_id's stream. There is no global event-log iteration, so read-model projections spanning many aggregates cannot be driven the pyfly way.
  - Where: pyfly src/pyfly/eventsourcing/{store.py (stream_all),projection.py (ProjectionRunner._loop)} vs rust crates/eventsourcing/src/{aggregate.rs (EventStore trait),sql_store.rs,projection.rs}

- **[container/aop/plugins/shell/websocket/cli/rule_engine/config_server/starters]** rule_engine: 4 leaf operators present in pyfly's evaluator are not implemented in the Rust Op enum — `between` (range check against [lo,hi]), `not_contains`, `exists` (field present and non-null), and `is_empty` (null/empty-string/empty-list/empty-map). The Rust Op enum is the Go-parity set (eq/ne/lt/lte/gt/gte/in/notIn/contains/startsWith/endsWith/matches/isNull/isNotNull) plus an `Op::Other(String)` fallback that fails evaluation with an unknown-op error. A migrating consumer whose YAML rules use any of these four operators would hit a hard evaluation failure. Verified: crates/rule-engine/src/models.rs Op enum + core.rs evaluator vs pyfly/rule_engine/evaluator.py _eval_condition.
  - Where: /Users/ancongui/Development/fireflyframework/fireflyframework-rust/crates/rule-engine/src/models.rs (Op enum ~L35-67) and /Users/ancongui/Development/fireflyframework/fireflyframework-rust/crates/rule-engine/src/core.rs

## MEDIUM

- **[kernel, config, context, domain, validation]** pyfly validation exposes is_valid_national_id / valid_national_id (5-20 alnum after normalization) and is_valid_tax_id / valid_tax_id (3-20 [A-Z0-9]). Neither validator exists anywhere in the Rust workspace. The validators crate ports CVV/PIN/amount/account-number/interest-rate/date/datetime but skips these two. A migrating consumer calling either predicate has no counterpart (note: validate_dni/nie/nif/ssn/vat are nation-SPECIFIC and do NOT cover pyfly's generic national-id/tax-id format checks).
  - Where: pyfly: src/pyfly/validation/domain.py (is_valid_national_id, is_valid_tax_id, valid_national_id, valid_tax_id). Rust: crates/validators/src/lib.rs has no national_id/tax_id fn (grep across crates/ returns nothing).

- **[data, cqrs, eventsourcing, transactional]** Runtime object-to-object mapper (MapStruct equivalent). pyfly data.mapper provides Mapper.map/map_list/project, register_projection, the @mapping decorator and default_mapper — name-matched DTO<->entity mapping with renaming, transformers, exclusion, nested-model recursion. No equivalent exists anywhere in the Rust workspace (grep for Mapper/map_into/to_dto finds nothing). Consumers porting MapStruct-style mappings must hand-roll From/Into.
  - Where: pyfly src/pyfly/data/mapper.py (Mapper, mapping, default_mapper) — no Rust counterpart

- **[data, cqrs, eventsourcing, transactional]** Spring-Data derived query methods and @query custom queries. pyfly data.query_parser (QueryMethodParser parsing find_by_/count_by_/exists_by_/delete_by_ with operator suffixes, _and_/_or_ connectors, order_by chains) and data.query (@query for SQL/JPQL or Mongo filter strings) let repositories declare queries by method name or annotation. The Rust data crate has no derived-query parser or @query equivalent; consumers must implement the Repository trait and write Filter/SQL by hand.
  - Where: pyfly src/pyfly/data/{query_parser.py,query.py,post_processor.py} — no Rust counterpart

- **[data, cqrs, eventsourcing, transactional]** Spring-style Pageable / Sort / Order request types. pyfly data.pageable exposes Pageable (page>=1 validation, of/unpaged, is_paged, offset, next/previous) and Sort/Order (by, asc/desc, unsorted, and_then, ascending/descending). Rust crates/data has Page (the result envelope) and a filter-level Sort{field,direction} but no Pageable request builder or Sort/Order composition API; paging is expressed as raw page/size usize on Filter.
  - Where: pyfly src/pyfly/data/pageable.py vs rust crates/data/src/{page.rs,filter.rs}

- **[data, cqrs, eventsourcing, transactional]** Event-store multi-tenancy. pyfly StoredEventEnvelope carries a first-class tenant_id field, persisted as a column by SqlAlchemyEventStore and forwarded as a header by EventSourcingPublisher. The Rust DomainEvent has no tenant_id field (only a generic metadata BTreeMap); the SqlEventStore schema has no tenant column. Tenant-scoped event sourcing must be improvised via metadata.
  - Where: pyfly src/pyfly/eventsourcing/event.py (StoredEventEnvelope.tenant_id) + store.py vs rust crates/eventsourcing/src/{aggregate.rs (DomainEvent),sql_store.rs (DDL)}

- **[data, cqrs, eventsourcing, transactional]** Generic EventSourcedRepository. pyfly eventsourcing.repository.EventSourcedRepository ties together load (apply snapshot then replay events from snapshot sequence) and save (append pending events + snapshot when crossing snapshot_interval). Rust ships the parts (AggregateRoot, EventStore, SnapshotStore) but no repository orchestrating snapshot-interval persistence and rehydration; consumers wire that glue themselves each time.
  - Where: pyfly src/pyfly/eventsourcing/repository.py — no Rust counterpart (crates/eventsourcing exposes only the building blocks)

- **[eda + messaging + cache]** EDA event filters (EventFilter / HeaderEventFilter / PredicateEventFilter) have no Rust counterpart. pyfly exposes a pluggable delivery-gate abstraction: a header-regex filter and an arbitrary-predicate filter that decide whether an envelope is delivered to a handler. The Rust eda crate dispatches purely on glob topic/type pattern matching (InMemoryBroker, all transports) with no predicate/header-value filtering hook. A consumer that relied on @event_listener filtering by header value or a custom predicate must re-implement it inside each handler body.
  - Where: pyfly: src/pyfly/eda/filter.py (EventFilter, HeaderEventFilter, PredicateEventFilter). Rust: absent — grep across crates/eda*, crates/messaging finds no EventFilter/accepts/predicate construct; crates/eda/src/inmemory.rs + ports.rs only match on glob topic patterns.

- **[eda + messaging + cache]** Queryable EDA dead-letter store (EdaDeadLetterStore / EdaDeadLetterEntry / InMemoryEdaDeadLetterStore) has no Rust counterpart. pyfly ships a store of failed *events* with add/list(limit)/delete for operator inspection and replay. Rust covers DLQ *routing* (wrap_listener republishes an exhausted event to a dead-letter topic with x-original-topic/x-exception headers) but provides no inspectable store of captured failed events. The orchestration crate's DeadLetterEntry/DeadLetterService is saga/workflow/TCC-execution-scoped (execution_name, step_id, replay input), not EDA-event-scoped, so it does not serve this use case. A consumer that listed/inspected/deleted dead-lettered events via EdaDeadLetterStore has no equivalent API.
  - Where: pyfly: src/pyfly/eda/dlq.py (EdaDeadLetterStore.add/list/delete, InMemoryEdaDeadLetterStore). Rust: crates/eda/src/listener.rs has topic-routing DLQ only; crates/orchestration/src/dlq.rs is execution-scoped (saga/workflow), not event-scoped.

  - Where: crates/actuator/src/handler.rs (env_handler) + crates/admin/src/router.rs (env/config handlers, lines 162-179); source available but unwired in crates/config/src/lib.rs (Layered::property_sources)

  - Where: /Users/ancongui/Development/fireflyframework/fireflyframework-rust/crates/rule-engine/src/core.rs and service.rs vs /Users/ancongui/Development/fireflyframework/fireflyframework-pyfly/src/pyfly/rule_engine/evaluator.py (EvaluationMode.ALL/FIRST_MATCH)

## LOW

- **[kernel, config, context, domain, validation]** pyfly's Environment.accepts_profiles(*exprs) implements the Spring Boot 2.4+ profile-expression grammar (negation '!prod', boolean operators with grouping '(prod & cloud) | qa', comma-OR). The Rust config crate has active_profiles / multi_profile_sources for loading profile-specific YAML, but no profile-EXPRESSION evaluator. A consumer that gates logic on 'prod & cloud' style expressions has no equivalent; only simple membership against the active list is available.
  - Where: pyfly: src/pyfly/context/environment.py (accepts_profiles, _eval_boolean_profile). Rust: crates/config/src/profile.rs (active_profiles only; no accepts/expression logic; grep for accepts_profile/profile.expression across crates/ returns nothing).

- **[kernel, config, context, domain, validation]** pyfly's in-process application event mechanism — ApplicationEventBus (isinstance-typed subscribe/publish, @order-sorted listeners), ApplicationEventPublisher (injectable), @app_event_listener, and the lifecycle events ContextRefreshedEvent / ApplicationReadyEvent / ContextClosedEvent — has no Rust counterpart. RefreshScopeRefreshedEvent's payload (evicted keys) is returned directly by RefreshScope::refresh(), but there is no synchronous in-process pub/sub bus for lifecycle or arbitrary domain events. firefly-eda's InMemoryBroker is the async transport-style fan-out, not the type-dispatched synchronous ApplicationEventBus; firefly-lifecycle provides ordered start/stop hooks but not event publication. A consumer relying on @app_event_listener-style decoupled handlers within the process must roll their own.
  - Where: pyfly: src/pyfly/context/events.py (ApplicationEventBus, ApplicationEventPublisher, ApplicationEvent subclasses, app_event_listener). Rust: no ApplicationEvent/EventBus/EventPublisher in crates/ (grep returns nothing outside eda/sse/websocket); crates/eda is async broker, crates/lifecycle is hooks-only.

- **[kernel, config, context, domain, validation]** pyfly kernel's typed ErrorResponse model (RFC 7807-inspired) carries first-class classification fields — ErrorCategory and ErrorSeverity enums, retryable, retry_after, field_errors (list[FieldError]), trace_id/span_id/transaction_id, suggestion, documentation_url — with a to_dict() that omits empty optionals. The Rust kernel ProblemDetail exposes only the standard RFC 7807 members plus a flat extensions map; ErrorCategory/ErrorSeverity/FieldError/retryable are not defined as typed constructs anywhere. Equivalent output is reachable by attaching extension members via ProblemDetail::with(...) (the kernel test sets 'retryable' that way), but there is no typed structured-error model nor the category/severity enums, and the emitted key naming/shape differs from pyfly's ErrorResponse.to_dict().
  - Where: pyfly: src/pyfly/kernel/types.py (ErrorCategory, ErrorSeverity, FieldError, ErrorResponse). Rust: crates/kernel/src/problem.rs (ProblemDetail = standard members + extensions only); no ErrorCategory/ErrorSeverity/FieldError type in crates/ (grep returns nothing).

The Rust web crate has no runtime-config equivalent — JSON shaping is per-struct serde attributes (#[serde(rename_all="camelCase")], skip_serializing_if) decided at compile time. A consumer migrating a service that flips camelCase/exclude-none/fail-on-unknown via a config key (or registers a global encoder for a non-serde type) has no drop-in.
