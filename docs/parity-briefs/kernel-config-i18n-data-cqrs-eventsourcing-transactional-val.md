# Gap report: pyfly `kernel+config+i18n+data+cqrs+eventsourcing+transactional+validation+testing+rule_engine+plugins+starters+config_server+callbacks+webhooks+resources (spot-check bundle)`

Verdict: **partial** · est. 5100 LOC

## What it is
The horizontal platform tier of pyfly: kernel error taxonomy + lifecycle, typed config binding with entry-point auto-configuration discovery, i18n message sources, Spring-Data-style repositories (SQLAlchemy + Beanie/Mongo) with derived query methods/specifications/auditing/soft-delete, a full CQRS bus (validation/authorization/caching/context/tracing), event sourcing with outbox+upcasters, a 12.8k-loc saga/workflow/TCC engine with durable persistence/recovery/DLQ/signals/timers, banking validators, pytest test slices + testcontainers helpers, a YAML rule engine, a plugin system with dependency resolution and extension points, layer starters, a Spring-Cloud-Config-compatible config server with fs/git backends, HMAC callback dispatch, and webhook signature validation with idempotency stores. Python-runtime machinery excluded from gap accounting: entry-point AutoConfiguration discovery, pydantic decorators (validate_input/validator), __getattr__-driven derived query-method names (Rust idiom = the existing Filter DSL), DI-container test slices/mock_bean, and ASGI test client (Rust idiom = tower::ServiceExt::oneshot).

## pyfly public surface
kernel: PyFlyException taxonomy (37 types), ErrorResponse/FieldError, Lifecycle. config: AutoConfiguration, discover_auto_configurations, properties dataclasses. i18n: MessageSource, ResourceBundleMessageSource, AcceptHeader/FixedLocaleResolver. data: Page/Pageable/Sort, RepositoryPort/CrudRepository/PagingRepository, Specification, QueryMethodParser, Mapper, projection, transactional(Propagation/Isolation), SQLAlchemy+Mongo repos with auditing/soft-delete/routing. cqrs: Command/Query, Command/QueryBus, HandlerRegistry, ExecutionContext, AuthorizationService, QueryCache+EDA invalidator, fluent builders, actuator endpoint. eventsourcing: AggregateRoot, DomainEvent, EventStore (memory+SQL), SnapshotStore, TransactionalOutbox, EventUpcaster, Projection(Runner). transactional: @saga/@workflow/@tcc engines, SagaEngine/WorkflowEngine(signals/timers/child/continue-as-new/queries)/TccEngine, ExecutionPersistenceProvider(memory/Redis/SQL), RecoveryService, DeadLetterService, EventGateway, OrchestrationScheduler/Validator/Metrics/Tracer, ExecutionReport. validation: 16 is_valid_* banking predicates + pydantic glue. testing: PyFlyTestClient, slices, mock_bean, testcontainers helpers, assert_event_published. rule_engine: Rule/RuleSet/Condition/Action DSL+YAML loader, RuleEvaluator, ActionHandler, RuleSetRepository, RuleEngineService, validate_ruleset. plugins: Plugin/PluginManager/PluginDependencyResolver/ExtensionRegistry/extension_point. starters: enable/register_{core,web,application,data,domain}_stack. config_server: ConfigServer, ConfigClient, ConfigBackend (Filesystem/InMemory/Git). callbacks: CallbackDispatcher, CallbackConfig/Subscription/Execution, AuthorizedDomain, repositories. webhooks: WebhookProcessor, SignatureValidator (Hmac/GitHub/Stripe/Twilio/NoOp), WebhookEventStore (memory/Redis). resources: package data only.

## Missing in Rust
Covered outright: kernel, config, i18n, starters, testkit, callbacks (minor: AuthorizedDomain allowlist), resources. Real gaps, verified in crates/*/src: (1) transactional engine — crates/orchestration (1.5k loc) runs only in-process saga/DAG-workflow/TCC with cancellation; zero matches for persistence/recovery/DLQ/signal/timer in src; pyfly additionally has ExecutionPersistenceProvider (in-memory/Redis/SQLAlchemy), RecoveryService, DeadLetterService, signal/timer/child-workflow/continue-as-new/query services, EventGateway+EventTrigger, scheduled sagas, ExecutionReport, OrchestrationValidator, REST controllers; crates/transactional only covers pyfly's data.transactional (with_tx). (2) cqrs — no authorization phase (AuthorizationResult/Service), no ExecutionContext-aware handlers, no EDA cache-invalidation bridge, no fluent command/query builders, no CQRS actuator health/metrics endpoint. (3) eventsourcing — no TransactionalOutbox, no EventUpcaster hook, no SQL-backed EventStore (README admits Postgres store is roadmap). (4) data — no Specification combinators, no auditing (auto created/updated stamps), no soft-delete helpers, no named-datasource/read-write routing, no SQL- or Mongo-backed repository impl anywhere (no sqlx/mongo in workspace deps); only Page+Filter+Repository trait+MemoryRepository. (5) config_server — read-only MemoryStore + router only: no FilesystemConfigBackend with tiered search locations, no GitConfigBackend, no save/write path, no client-side ConfigClient (nothing in crates/config fetches remote propertySources). (6) plugins — Registry is registration-order only: no depends_on topological resolution (PluginDependencyResolver), no extension points/ExtensionRegistry, no per-plugin state machine. (7) rule_engine — AstEvaluator returns a Verdict with the action list but has no ActionHandler execution (set/increment/log applied to output facts), no RuleSetRepository/RuleEngineService for named-ruleset management. (8) validators — missing is_valid_cvv, is_valid_pin, is_valid_amount, is_valid_account_number, is_valid_interest_rate, is_valid_date/datetime (Rust has extras pyfly lacks: email, VAT, DNI/NIE/NIF, SSN, BSB). (9) webhooks — no idempotency WebhookEventStore (already_processed/remember) in the pipeline; pyfly has memory+Redis impls.

## Porting plan
All gaps are extensions of existing crates; no new crate needed. (1) crates/orchestration (~2,800 loc): add ExecutionState/StepStatus/ExecutionReport models; trait PersistenceProvider {save/load/list/delete} + MemoryPersistence; RecoveryService (scan stuck executions, resume/compensate); trait DeadLetterStore + entry model; SignalService (tokio::sync::{watch,oneshot} keyed by correlation id) with Workflow::wait_for_signal nodes; TimerService via tokio::time::sleep nodes; EventGateway port + TriggerMode for broker-driven saga starts (reuse firefly-eda Broker); axum router for executions/signals (mirror callbacks/web.rs pattern). Decorators (@saga_step etc.) stay builder methods. Redis/SQL persistence adapters: define the port only, ship memory impl — workspace has no redis/sqlx dep; follow the existing typed-config Stub adapter policy or add rusqlite-backed impl reusing the dev-only dep as in migrations. (2) crates/cqrs (~450 loc): AuthorizationMiddleware + Message::authorize(ctx) default method (same pattern as validate); ExecutionContext struct threaded via Envelope extensions; EdaInvalidationBridge subscribing CacheInvalidationEvent on firefly-eda Broker calling QueryCache::invalidate; fluent CommandBuilder/QueryBuilder. (3) crates/eventsourcing (~400 loc): trait Upcaster {upcast(DomainEvent)->DomainEvent} applied on load; OutboxRecord + TransactionalOutbox polling publisher over firefly-eda; SQL EventStore over the firefly-transactional Database port (rusqlite-tested). (4) crates/data (~450 loc): Specification<T> enum (And/Or/Not/Pred) lowering to Filter; AuditStamps helper; soft_delete predicate injection on Filter; NamedDataSources registry + RoutingPolicy (read/write split) as plain structs. (5) crates/config-server (~400 loc): FsStore (tiered dirs, serde_yaml/json) + GitStore (shell out to git binary, then delegate to FsStore — matches pyfly's adapter); add optional save() to Store; ConfigClient in crates/config: reqwest fetch of /{app}/{profile}/{label}, flatten propertySources into a config Source. (6) crates/plugins (~200 loc): depends_on() on Plugin trait, Kahn topological start order with cycle error, PluginState enum tracking, ExtensionRegistry (TypeId-keyed like cqrs Bus). (7) crates/rule-engine (~250 loc): trait ActionHandler {apply(action, fact)->fact}, builtin set/increment/log handlers, RuleSetRepository trait + MemoryRuleSetRepository + RuleEngineService (evaluate_by_name), wire into web.rs. (8) crates/validators (~100 loc): validate_cvv/pin/amount/account_number/interest_rate/date(chrono parse). (9) crates/webhooks (~80 loc): trait EventStore {already_processed/remember} + MemoryEventStore, dedup check in Pipeline before processor dispatch.
