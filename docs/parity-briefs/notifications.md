# Gap report: pyfly `notifications`

Verdict: **partial** · est. 3800 LOC

## What it is
pyfly.notifications is the email/SMS/push abstraction with hexagonal provider adapters. It ships rich channel-specific message models (EmailMessage with cc/bcc, text+html bodies, attachments, custom headers, template routing; SmsMessage; PushMessage), default services that add per-recipient opt-out suppression (SUPPRESSED status), local Jinja2 template rendering with provider-native-template precedence, sent/failed/suppressed metric counters, and exception-to-FAILED-result conversion. Critically, all five vendor providers are FULL working implementations — SMTP (stdlib smtplib via thread executor, verified against a real in-process aiosmtpd server), SendGrid v3 /mail/send, Resend /emails, Twilio Messages.json, and Firebase FCM v1 (per-token with partial-success semantics) — plus in-memory Dummy providers; auto_configuration.py is Python DI/bean wiring (excluded from the gap), though its config-keyed provider selection is portable as from_config constructors.

## pyfly public surface
Models: EmailStatus (QUEUED/SENT/DELIVERED/BOUNCED/FAILED/SUPPRESSED), Attachment, EmailMessage, SmsMessage, PushMessage, NotificationResult. Ports: EmailProvider/SmsProvider/PushProvider, EmailService/SmsService/PushService. Services: DefaultEmailService/DefaultSmsService/DefaultPushService (opt-out pruning, template precedence, metrics, error-to-FAILED). Template: NotificationTemplateEngine, Jinja2TemplateEngine, NoOpTemplateEngine. Preferences: NotificationPreferenceService, InMemoryPreferenceService (opt_out/opt_in/is_opted_in with recipient normalization). Providers: SmtpEmailProvider, SendGridEmailProvider, ResendEmailProvider, TwilioSmsProvider, FirebasePushProvider, DummyEmail/Sms/PushProvider. Plus NotificationsAutoConfiguration (DI wiring).

## Missing in Rust
Rust firefly-notifications has only the thin Go-parity core: Notification envelope (single to, single body), Kind, Channel trait, Dispatcher, MemoryChannel. Missing: (1) ALL real vendor delivery — the four crates notifications-sendgrid/-resend/-twilio/-firebase are documented contract-only stubs whose send() always returns ERR_NOT_IMPLEMENTED, while pyfly's are full HTTP integrations; (2) SMTP email adapter — no counterpart anywhere in the workspace (no lettre dep, no smtp code); (3) rich email semantics — cc/bcc, attachments, custom headers, separate text/html bodies, provider message IDs, delivery-status enum and NotificationResult; (4) opt-out/preference service with per-recipient pruning and SUPPRESSED short-circuit (incl. recipient normalization); (5) local template rendering engine and the local-vs-provider-native precedence rule; (6) sent/failed/suppressed metric counters in the send path; (7) config-driven provider selection (sendgrid/resend/smtp/twilio/firebase/dummy from config keys). Excluded as Python-runtime machinery: lazy httpx imports with pip-extra error messages, asyncio run_in_executor bridging for blocking smtplib, runtime_checkable Protocols, and the @bean/@auto_configuration container itself.

## Porting plan
Extend crates/notifications plus the four existing vendor crates, and add one new crate firefly-notifications-smtp. (a) In firefly-notifications add rich types alongside the Go-parity envelope (keep Notification/Dispatcher untouched for Go wire parity): EmailMessage/SmsMessage/PushMessage, Attachment, DeliveryStatus enum, NotificationResult; async_trait ports EmailProvider/SmsProvider/PushProvider; EmailService/SmsService/PushService orchestrators implementing opt-out pruning (to+cc+bcc, per device token), SUPPRESSED short-circuit, template precedence, and provider-error-to-FAILED-result conversion (idiom: services return Ok(NotificationResult) and fold provider Err into status=FAILED, matching pyfly's _send_safely); PreferenceService trait + InMemoryPreferenceService (RwLock<HashSet<(String,String)>> with email-lowercase/phone-digit normalization); TemplateEngine trait + a feature-gated engine (add minijinja or tera to the workspace catalog; a NoOp engine as default); metric hooks as an injected trait object wired to firefly-actuator's MetricRegistry counters (labels channel/provider). Replace pyfly's @bean wiring with explicit from_config(&Config) constructors per provider (registry/builder idiom). (b) Implement real send() in the four vendor crates using workspace reqwest 0.12 (json + form for Twilio basic-auth form post) and base64 0.22 for attachments: SendGrid v3 personalizations + dynamic_template_data + X-Message-Id; Resend /emails payload; Twilio Messages.json with sender-precedence and missing-sender error; Firebase FCM v1 per-token loop with partial-success result. Keep the existing stub Channel/sentinels for Go parity and add the rich-provider impls beside them (each crate also impls notifications::Channel by mapping the thin envelope). (c) New crate firefly-notifications-smtp using lettre (add to catalog, tokio1 + rustls features): STARTTLS, auth, multipart text/html, attachments, custom non-reserved headers, bcc not leaked in headers. Tests: behavior tests against an axum-based mock HTTP server (axum already in catalog) mirroring tests/notifications/test_*_behavior.py, plus opt-out/template/metrics tests mirroring test_template_and_preferences.py and test_optout_per_recipient.py. Estimated source LOC ~1,900 (core extension ~800, four vendor integrations ~700, smtp crate ~300, config plumbing ~100); roughly the same again in tests (~3,800 total).
